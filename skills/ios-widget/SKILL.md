# iOS Widget — WidgetKit Specialist

Build a complete WidgetKit widget from specification: timeline provider, views for all relevant sizes, App Group data sharing, and interactive elements.

## Process

### Step 1: Read Widget Specification

Read widget requirements from one of:
- `docs/product-vision/widget-spec.md` (if generated by ios-prd)
- `$ARGUMENTS` if user provided an inline description
- Analyze existing app code to infer appropriate widget content

Determine:
- What data does the widget display?
- Which size families make sense?
- Does the user need to configure the widget (pick an item, choose a setting)?
- Does it need interactive elements (button, toggle)?
- Is it time-sensitive (countdown) or data-driven (daily content)?

### Step 2: Fetch Current WidgetKit Docs

Before writing any code, use context7 MCP:
1. Use `resolve-library-id` to find "WidgetKit apple developer"
2. Query for `AppIntentTimelineProvider` and `TimelineProvider` differences
3. Query for interactive widget patterns (`Button`, `Toggle` in widget views)
4. Query for `WidgetFamily` and size specifications
5. Check for any deprecations (e.g., `IntentTimelineProvider` is superseded by `AppIntentTimelineProvider`)

Load the local reference as well: `${CLAUDE_SKILL_DIR}/widgetkit.md`

### Step 3: Choose Provider Type

**Decision tree:**

```
Does the user configure which data the widget shows?
  YES → AppIntentTimelineProvider (configurable widget)
  NO  → TimelineProvider (static widget, one configuration)
```

`AppIntentTimelineProvider` uses an `AppIntent` struct to define configuration options. The system shows a configuration sheet when the user long-presses the widget.

`TimelineProvider` is simpler — no user configuration, just generates entries.

### Step 4: Design Timeline Entry

```swift
struct WidgetEntry: TimelineEntry {
    let date: Date
    // All data the view needs — fetch at provider time, not in view
    let title: String
    let subtitle: String
    let accentColor: Color
    // Include a placeholder state flag
    var isPlaceholder: Bool = false
}
```

The entry must contain all data the view needs — no network calls or heavy computation in the view.

### Step 5: Implement Timeline Provider

**For `AppIntentTimelineProvider`:**

```swift
struct WidgetProvider: AppIntentTimelineProvider {
    typealias Intent = WidgetConfigIntent  // Your AppIntent
    typealias Entry = WidgetEntry
    
    func placeholder(in context: Context) -> WidgetEntry {
        WidgetEntry(date: .now, title: "Loading...", subtitle: "", isPlaceholder: true)
    }
    
    func snapshot(for configuration: WidgetConfigIntent, in context: Context) async -> WidgetEntry {
        // Return quickly — system uses this for gallery preview
        await fetchEntry(for: configuration)
    }
    
    func timeline(for configuration: WidgetConfigIntent, in context: Context) async -> Timeline<WidgetEntry> {
        let entry = await fetchEntry(for: configuration)
        
        // Reload policy:
        // .after(date) — reload at specific future time (for countdowns, time-sensitive)
        // .atEnd — reload after last entry expires (for daily content)
        // .never — don't reload until app signals (WidgetCenter.shared.reloadTimelines)
        
        let nextRefresh = Calendar.current.date(byAdding: .hour, value: 1, to: .now)!
        return Timeline(entries: [entry], policy: .after(nextRefresh))
    }
    
    private func fetchEntry(for configuration: WidgetConfigIntent) async -> WidgetEntry {
        // Read from App Group shared data — NOT network calls
        let sharedDefaults = UserDefaults(suiteName: "group.com.yourapp.shared")
        // ... read cached data
        return WidgetEntry(date: .now, title: "Title", subtitle: "Subtitle")
    }
}
```

**Refresh budget guidance (document in comments):**
- The system provides ~40-70 reloads/day per widget (varies by usage patterns)
- Reloads requested more frequently than the budget allows are silently deferred
- Never assume an exact reload time — design UI to degrade gracefully if data is slightly stale
- Use `WidgetCenter.shared.reloadTimelines(ofKind:)` from the main app when data changes

### Step 6: Generate Widget Views

Create a view for each supported family, using `@Environment(\.widgetFamily)` to branch:

```swift
struct WidgetEntryView: View {
    var entry: WidgetEntry
    @Environment(\.widgetFamily) var family
    
    var body: some View {
        switch family {
        case .systemSmall:
            SmallWidgetView(entry: entry)
        case .systemMedium:
            MediumWidgetView(entry: entry)
        case .systemLarge:
            LargeWidgetView(entry: entry)
        case .accessoryCircular:
            AccessoryCircularView(entry: entry)
        case .accessoryRectangular:
            AccessoryRectangularView(entry: entry)
        case .accessoryInline:
            AccessoryInlineView(entry: entry)
        default:
            SmallWidgetView(entry: entry)
        }
    }
}
```

Design principles per size:
- **systemSmall (158×158pt):** Single piece of glanceable info. No text below 11pt. One tap target at most.
- **systemMedium (338×158pt):** Two columns or a list of 2-3 items. Tappable rows.
- **systemLarge (338×354pt):** Full content. List of 5-6 items or rich detail.
- **accessoryCircular (44pt diameter):** Icon + number or progress ring only.
- **accessoryRectangular (~157×52pt):** 2-3 lines of text or small chart.
- **accessoryInline (~288×18pt):** Single line of text + optional icon.

### Step 7: Set Up App Group Data Sharing

The app and widget must share data via an App Group:

**Lightweight data (UserDefaults):**

```swift
// In main app — write when data changes
let defaults = UserDefaults(suiteName: "group.com.yourapp.shared")!
defaults.set(encodedData, forKey: "widgetData")
// Signal widget to refresh
WidgetCenter.shared.reloadTimelines(ofKind: "YourWidgetKind")
```

**SwiftData (complex structured data):**

```swift
// Shared ModelContainer — same config in both app and widget
let groupURL = FileManager.default
    .containerURL(forSecurityApplicationGroupIdentifier: "group.com.yourapp.shared")!
    .appendingPathComponent("data.sqlite")
let config = ModelConfiguration(url: groupURL)
let container = try ModelContainer(for: Item.self, configurations: [config])
```

See `ios-data-model` skill for full ModelContainer setup.

### Step 8: Add Interactive Elements (iOS 17+)

For buttons and toggles that trigger app actions:

```swift
// Define the AppIntent that the button triggers
struct CompleteItemIntent: AppIntent {
    static var title: LocalizedStringResource = "Complete Item"
    
    @Parameter(title: "Item ID") var itemID: String
    
    func perform() async throws -> some IntentResult {
        // Update shared data store
        // WidgetCenter.shared.reloadTimelines is called automatically
        return .result()
    }
}

// Use in widget view
struct MediumWidgetView: View {
    var entry: WidgetEntry
    
    var body: some View {
        VStack {
            Text(entry.title)
            Button(intent: CompleteItemIntent(itemID: entry.itemID)) {
                Label("Complete", systemImage: "checkmark.circle")
            }
            .buttonStyle(.plain)
        }
        .containerBackground(.fill.tertiary, for: .widget)
    }
}
```

Note: Interactive widgets require iOS 17+. Gate with `#available(iOS 17, *)` if supporting iOS 16.

### Step 9: Add Widget to Project

In the widget extension's main entry point:

```swift
@main
struct AppWidgets: WidgetBundle {
    var body: some Widget {
        MainWidget()
        // Add more widget types here
    }
}

struct MainWidget: Widget {
    let kind: String = "MainWidget"
    
    var body: some WidgetConfiguration {
        AppIntentConfiguration(
            kind: kind,
            intent: WidgetConfigIntent.self,
            provider: WidgetProvider()
        ) { entry in
            WidgetEntryView(entry: entry)
                .containerBackground(.fill.tertiary, for: .widget)
        }
        .configurationDisplayName("Widget Name")
        .description("What this widget shows.")
        .supportedFamilies([.systemSmall, .systemMedium, .systemLarge])
    }
}
```

### Step 10: TDD for Timeline Logic

Test the timeline provider logic in isolation:

```swift
@Test func timelineGeneratesCorrectEntries() async throws {
    let provider = WidgetProvider()
    let intent = WidgetConfigIntent()
    let context = TimelineProviderContext.preview(family: .systemMedium)
    
    let timeline = await provider.timeline(for: intent, in: context)
    
    #expect(!timeline.entries.isEmpty)
    #expect(timeline.policy != .never)
}
```

Focus tests on:
- Entry date calculation (are entries at the right times?)
- Reload policy selection (does time-sensitive content use `.after`?)
- Data reading from shared container
- Placeholder/snapshot state correctness
